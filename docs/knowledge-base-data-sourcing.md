# 知识库：数据来源与进库治理

> 版本：2026-05-19  
> 关联：[分层品种 / 病害 / 环境库与搜索体系](./knowledge-base-layered-search-design.md)

本文回答：**知识库数据从哪来、怎么进库、怎么长期可信**。技术分层解决「放哪、怎么搜」；数据问题解决「**谁为正确性负责**」。

---

## 1. 问题本质

没有单一银弹：**L1 品种**、**L2 病虫害**、**L3 环境叙事** 的可信来源、更新频率和版权约束不同，通常需要 **组合供给**。

与当前仓库的关系简述：

- **品种要点**：已是「百科弱信号 + LLM 推断 + `SpeciesProfile` 缓存」；瓶颈在 **质量与版权边界**，不仅是表结构。
- **诊断症状**：`diagnoseEngine` 内固定枚举 + 规则；扩展依赖 **规则/词条生产** 或 **外采文库**。
- **百科式长文**：`miniprogram/data/knowledge.js` 等静态资产，来源只能是 **自建或采购导入**。

---

## 2. 数据可以从哪来

| 来源 | 适合承载 | 优点 | 主要风险 |
|------|-----------|------|----------|
| **自建 / 写手 / 合作园艺师** | L2 长文、L3 场景文、争议品种要点 | 可控、可签版权、语气统一 | 成本高、扩容慢 |
| **识别链上的弱信号**（如第三方百科摘要、学名/俗名） | L1 上下文、LLM 输入 | 不额外采购；与识别闭环近 | **转载与 API 条款**须合规；摘要噪声大 |
| **LLM 生成 + 人工审核** | 初稿、缺条补全 | 快、覆盖面大 | 幻觉、过时；须 **draft → publish** 与免责声明 |
| **结构化开放数据**（如分类库映射） | L0 拉丁名 / 分类 ID | 可机器对齐 | 俗名、家庭园艺语境仍要自建别名与文案 |
| **UGC / 社区** | 技巧、地区经验 | 长尾鲜活 | 审核与舆情、法律条款要前置 |
| **采购版权库 / 出版社** | 图鉴级正文 | 授权清晰 | 商务与格式清洗成本 |

---

## 3. 建议的进库流水线（与来源解耦）

无论数据从哪来，统一纪律，便于换供应商、换模型而不乱库：

1. **采集元数据**：`sourceType`（如 `manual` | `partner` | `llm_draft` | `user` | `api_snippet`）+ `sourceRef`（链接、合同编号、模型名与版本）。
2. **归一**：品种过 **`nameKey`**（与现有实现一致）；病害挂稳定 **`symptomId` / `slug`**；避免只有长文、没有锚点。
3. **审核态**：LLM 与爬虫摘要默认 **draft**；**publish** 需人工或规则集（禁用词、长度、是否含不当医疗承诺）。
4. **版本与审计**：`disclaimerVersion`、`contentHash`；重大修改可追溯；必要时 `KnowledgeEditLog`。
5. **下线策略**：侵权、纠错、过时 → `deprecatedAt` + 指向替代篇，避免硬删断链。

（具体表字段可在实现分层库时落到 Prisma，本文只定原则。）

---

## 4. MVP 阶段最务实的组合

在「人不能无限写、又不能全靠模型」之间取平衡：

- **L1 品种**：继续 **识别/百科片段 + LLM + 自建表**；对 **高频品种** 逐步人工定稿为 `manual`（或等价「已审」标记），投入小、体验提升明显。
- **L2 病虫害**：**少而精** —— 先 10～20 篇高频「症状 → 处置」词条（内部或顾问），规则诊断只链到这些；不追求第一天百科全量。
- **L3 环境**：从 **`careEngine` / `PlantEnv` 已有维度** 反写少量短文（暖气房、空调房、南向窗台等），与引擎语义对齐，**外部版权风险低**。

**第一作者**明确为：团队 + 顾问（必要时采购），再逐步加 UGC 或更大文库。

---

## 5. 合规与产品表述

- 诊断与订阅类文案：避免「包治」「实验室级确诊」等表述；词条与模板统一 **免责声明版本**。
- 使用第三方摘要：以 **合同与 robots/开发者条款** 为准，控制摘要长度与展示形态，必要时仅作 **离线推理输入** 不落全文。

---

## 6. 一句话结论

**最大问题不是「存哪」，而是「谁为正确性付费」**：人写/人审、买版权，或接受 **LLM + 强免责声明 + 快速纠错** 并 **控制范围**。  
技术上已有「弱事实 + 模型 + 自建缓存」的品种路径；知识库要做大，必须把 **source、审核态、版权与下线** 写成硬规范，否则搜索与分层设计只是空壳或合规雷区。
