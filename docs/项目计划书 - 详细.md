## 一、项目概述

### 1.1 核心定位

打造家庭场景最专业的**智能植物管家**，通过 AI + 硬件深度本地化组合，帮助城市家庭用户解决"植物买回家养两天就死"的根本问题。

### 1.2 三年愿景

| 时间  | 目标                   |
| --- | -------------------- |
| 第1年 | 验证产品，50万用户，3%付费率     |
| 第2年 | 品类第一，300万用户，8%付费率    |
| 第3年 | 市场领导者，1000万用户，12%付费率 |

### 1.3 核心价值主张

**不是给你一个通用的"绿植浇水指南"，而是告诉你"你家的植物在你的环境下应该怎么养"。**

### 1.4 需求验证

| 诊断维度  | 验证结果                           |
| ----- | ------------------------------ |
| 需求真实性 | ✅ 创始人每年养死5-10棵植物，老婆/同事同样如此     |
| 当前解法  | 搜索/问AI，但找到答案后仍然救不活             |
| 核心问题  | 信息无法落地 — 缺少针对"我的植物+我的环境"的个性化方案 |
| 目标用户  | 年轻上班族/城市家庭                     |
| 产品优先级 | 1.智能提醒 2.在线诊断 3.植物识别           |
| 趋势判断  | 有利 — AI技术成熟+城市化+年轻人忙碌          |

## 二、市场分析

### 2.1 市场规模

#### 中国室内植物市场

| 指标       | 数据        | 来源/估算     |
| -------- | --------- | --------- |
| 室内植物年销量  | 约10亿盆     | 行业协会估算    |
| 家庭园艺用户   | 约3-4亿家庭   | 电商平台数据    |
| 客单价      | 20-100元/盆 | 天猫/京东数据   |
| 年市场规模    | 约500亿     | 按销量×均价估算  |
| 每年植物死亡损失 | 估计数百亿     | 按70%死亡率估算 |

#### 智能植物养护市场

| 指标     | 数据            | 说明    |
| ------ | ------------- | ----- |
| 当前渗透率  | <1%           | 蓝海市场  |
| 预测市场规模 | 50-100亿       | 2028年 |
| 增长驱动   | AI技术成熟+硬件成本下降 | 长期趋势  |

### 2.2 用户痛点

#### 痛点描述

1. **植物死亡率高**：买回家2天就死，70%以上在90天内死亡
2. **养护知识缺乏**：不知道植物叫什么、应该怎么养
3. **问题无法解决**：遇到问题搜索/问AI，但答案无法落地
4. **环境差异大**：不同城市、不同季节、不同朝向，养护方式完全不同

#### 用户画像

| 维度   | 描述                        |
| ---- | ------------------------- |
| 核心用户 | 25-40岁城市白领/上班族            |
| 特征   | 喜欢植物但养不活、愿意为知识付费、熟练使用智能设备 |
| 痛点   | 没时间、没经验、租房环境不稳定           |
| 需求   | 简单、自动化、不需要学习的植物养护方案       |

### 2.3 市场趋势

#### 有利因素

1. **AI 技术成熟**：视觉识别准确率>95%，大模型对话能力产品化
2. **硬件成本下降**：传感器、芯片成本持续降低，IoT普及
3. **城市化+小户型**：室内植物需求增加，养护时间减少
4. **社交媒体热度**：小红书/抖音"植物杀手"话题持续火热
5. **消费升级**：愿意为宠物/植物买单的年轻人越来越多

#### 不利因素

1. **获客成本**：C端付费意愿需要培育
2. **复购率**：植物是非高频消费品
3. **竞争加剧**：大厂可能切入

### 3.2 核心功能详解

#### 3.2.1 植物识别

**用户场景**：
- 朋友送了一盆植物，不知道叫什么
- 路边看到好看的植物，想知道名字

**实现方式**：
1. 用户拍照上传
2. 调用植物识别API（百度/阿里植物识别）
3. 返回植物名称、科属、养护难度
4. 自动关联养护知识库

**差异化**：不是简单返回植物信息，而是结合用户环境给出养护建议。

#### 3.2.2 土壤识别

**用户场景**：
- 不确定什么时候该浇水
- 想确认土壤是否已经干透

**实现方式**：
1. 用户拍照土壤（花盆表面或插入土壤）
2. AI 图像分析土壤颜色和纹理
3. 判断土壤湿度等级：非常湿/偏湿/适中/偏干/非常干
4. 结合养护计划，给出是否应该浇水的建议

**技术方案**：
- **图像特征**：颜色（深浅）、纹理（裂纹）、反光度
- **模型**：百度图像识别/自研分类模型
- **准确率**：目测判断 70-80%（配合用户反馈优化）

**差异化**：不依赖硬件传感器，用软件方式低成本实现

---

#### 3.2.3 环境感知

**用户场景**：
- 不同城市、不同季节，养护方式完全不同
- 室内有暖气/空调，和室外不一样

**实现方式**：
1. **定位获取城市**：用户首次使用时请求定位权限
2. **天气API**：调用百度天气/心知天气API，获取：
   - 实时温度
   - 湿度
   - 未来3天预报
3. **用户设置**：室内/室外、有无暖气/空调、放置位置（朝南/朝北）
4. **动态调整**：
   - 北京冬天有暖气 → 空气干燥 → 增加浇水频率
   - 南方梅雨季 → 湿度高 → 减少浇水
   - 夏天高温 → 调整到早晨/晚上浇水

**API 成本**：
- 百度天气API：免费额度 1000次/天
- 心知天气：免费版足够中小规模使用

**差异化**：现有产品没有结合真实天气数据做动态调整

---

#### 3.2.4 个性化养护计划

**用户场景**：
- 知道植物叫什么，但不知道具体怎么养
- 网上搜到的方法不适合自己的环境

**实现方式**：
1. 用户添加植物（识别或手动选择）
2. 询问用户环境：城市、光照、放置位置、季节、有无暖气
3. 结合土壤状态、天气数据，生成个性化养护计划：
   - 浇水频率和量
   - 施肥时间和种类
   - 换盆时机
   - 注意事项
4. 计划可视化管理

**差异化**：不是给一个通用指南，而是"你家的植物在你的环境下应该怎么做" + "根据今天天气和土壤状况，应该现在浇水"。

---

#### 3.2.5 环境感知智能提醒

**用户场景**：
- 夏季和冬季的浇水频率完全不同
- 晴天和雨天应该差异化提醒
- 室内有暖气时提醒要补水

**实现方式**：
1. 基础提醒：根据养护计划生成固定提醒
2. 环境调整：根据天气API动态调整提醒时间
   - 高温天：提前提醒
   - 下雨天：延迟提醒
   - 暖气季：增加提醒频次
3. 土壤辅助：用户拍照土壤后，可以跳过/提前提醒
4. 用户反馈学习：用户跳过提醒 → 自动延长下次提醒

**核心算法**：
```
建议浇水时间 = 基础间隔 × 温度系数 × 湿度系数 × 土壤系数 × 季节系数
```

**差异化**：不是固定Schedule，而是动态感知的智能提醒

#### 3.2.6 环境感知智能提醒

**用户场景**：
- 忘记浇水/施肥
- 不确定什么时候该浇水
- 夏天和冬天应该不同提醒

**实现方式**：
1. 根据养护计划生成基础提醒任务
2. 结合天气API动态调整提醒时机
3. 用户拍照土壤后可以提前/延迟提醒
4. 用户反馈（跳过/完成）自动优化

**提醒类型**：
- 浇水提醒（动态调整）
- 施肥提醒
- 换盆提醒
- 病虫害检查提醒

**核心逻辑**：
```
提醒分数 = 基础分数 - 温度调整 - 湿度调整 + 土壤调整 + 季节调整
如果分数 > 阈值 → 触发提醒
```

#### 3.2.7 在线诊断

**用户场景**：
- 植物叶子黄了、蔫了、生虫了
- 不知道植物出了什么问题

**实现方式**：
1. 用户拍照并描述问题
2. AI分析图片+症状描述
3. 给出诊断结果和救治方案
4. 如无法确定，提供专家咨询入口

**诊断能力**：
- 病害识别（叶斑病、根腐病等）
- 虫害识别（蚜虫、红蜘蛛等）
- 环境问题（浇水过多/过少、光照问题等）
- 营养缺乏（缺氮、缺铁等）

### 3.3 交互设计

#### 核心流程

```
添加植物 → 识别品种 → 环境设置 → 生成计划 → 日常提醒 → 问题诊断
```

#### 首页设计

1. **我的植物**：展示用户添加的所有植物
2. **今日任务**：今天需要做的事情（浇水/施肥等）
3. **发现**：植物百科、养护知识
4. **我的**：个人中心、设置

## 四、技术方案

### 4.1 技术架构

#### 整体架构

```
┌─────────────────────────────────────────────────────────────┐
│                        用户层 (C端)                          │
│  微信小程序 | App (iOS/Android) | Web                       │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                        服务层                               │
│  API Gateway → 业务服务 → 消息队列 → 定时任务                │
│  (认证/用户/植物/提醒/诊断/硬件管理)                          │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                        数据层                               │
│  MySQL (业务数据) | Redis (缓存) | 对象存储 (图片)           │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                        AI 能力层                            │
│  植物识别API | 病害识别API | 大模型对话 | 养护建议生成        │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                        硬件层 (v2)                          │
│  智能花盆 → WiFi/蓝牙 → 设备接入服务                          │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 技术选型

| 层级  | 技术选型                 | 理由          |
| --- | -------------------- | ----------- |
| 前端  | 微信小程序 + React Native | 快速触达用户，跨平台  |
| 后端  | Node.js/Go + MySQL   | 快速开发，高并发    |
| AI  | 百度AI/阿里云 + 自研模型      | 成熟API + 定制化 |
| 硬件  | ESP32 + WiFi         | 低成本，易开发     |
| 云   | 阿里云/腾讯云              | 国内稳定，价格优惠   |

### 4.3 核心模块设计

#### 4.3.1 植物识别模块

**流程**：
1. 用户拍照上传
2. 预处理（缩放、裁剪）
3. 调用植物识别API
4. 结果解析和映射到知识库
5. 返回识别结果+养护信息

**API 选择**：
- 百度植物识别：免费额度高，准确率较好
- 阿里植物识别：品类更全
- 备选：自建模型（长期）

#### 4.3.2 土壤识别模块

**流程**：
1. 用户拍照土壤（支持直接拍摄或从相册选择）
2. 图像预处理（裁剪土壤区域、归一化）
3. 调用图像分类模型
4. 判断湿度等级（5档：非常湿/偏湿/适中/偏干/非常干）
5. 返回湿度状态 + 建议操作

**技术方案**：
```
用户拍照 → 土壤区域检测 → 颜色特征提取 → 分类模型 → 湿度判断 → 建议
```

**模型方案**：
- **短期**：使用百度图像识别API，训练一个简单的分类器
- **长期**：自建模型，使用公开数据集 + 用户反馈数据

**特征提取**：
- 颜色特征：RGB/HSV 颜色直方图
- 纹理特征：GLCM 纹理分析
- 组合特征：深度学习 CNN 特征

**数据来源**：
- 公开数据集：Plant Village 等
- 用户反馈：用户标记"土壤实际干湿程度"

#### 4.3.3 环境感知模块

**数据来源**：
1. **定位**：微信小程序定位 API → 城市名称
2. **天气API**：百度天气 API / 心知天气
   - 请求参数：城市名称
   - 返回数据：温度、湿度、风力、天气现象
3. **用户设置**：
   - 室内/室外
   - 有无暖气/空调
   - 放置位置（朝南/朝北/客厅/卧室/阳台）
   - 光照强度（强/中/弱）

**数据更新策略**：
- 天气数据：每天更新一次，缓存到 Redis
- 用户设置：变更时更新
- 提醒计算：每次提醒前重新计算

**API 调用示例**（百度天气）：
```json
{
  "city": "北京",
  "temperature": "-3",
  "humidity": "30",
  "weather": "晴",
  "forecast": [
    {"date": "今天", "temp": "-3~5", "weather": "晴"},
    {"date": "明天", "temp": "-5~2", "weather": "多云"}
  ]
}
```

**成本估算**：
- 免费额度：1000次/天
- 超出费用：约 0.1元/1000次
- 50万用户 × 1次/天 = 50万次/天 → 需要付费但成本可控

#### 4.3.4 养护计划生成模块

**输入**：
- 植物品种（品种数据库）
- 用户所在城市（位置）
- 实时天气数据（温度、湿度）
- 用户设置（室内/室外、光照、暖气）
- 土壤状态（如果用户拍照过）
- 季节

**输出**：
- 基础浇水频率（天数）
- 环境调整后的实际浇水频率
- 每次浇水量
- 施肥频率和种类
- 换盆建议
- 注意事项

**实现**：
- 规则引擎：基础养护逻辑（品种×环境矩阵）
- AI 优化：根据用户反馈和土壤数据调整参数
- 知识库：专家经验沉淀

**核心算法**：
```
实际浇水间隔 = 基础间隔 × 温度系数 × 湿度系数 × 土壤系数 × 位置系数

温度系数：高温天 → 系数 < 1 (缩短间隔)
湿度系数：雨天 → 系数 > 1 (延长间隔)
土壤系数：土壤偏干 → 系数 < 1 (提前提醒)
位置系数：朝南阳台 → 系数 < 1 (更频繁)
```

#### 4.3.5 环境感知智能提醒模块

**架构**：
```
养护计划 → 环境调整 → 提醒任务队列 → 定时触发 → 推送服务 → 用户手机
     ↓                    ↓
   天气API ←─────── 用户操作确认 → 更新下次提醒时间
     ↓
  土壤数据（可选）
```

**核心逻辑**：

1. **基础调度**：根据养护计划生成固定的任务队列
2. **环境预检查**：
   - 检查当天天气数据
   - 如果是高温/雨天，动态调整触发时间
3. **触发前检查**：
   - 如果用户有土壤拍照记录，检查土壤状态
   - 土壤偏湿 → 跳过本次提醒
   - 土壤偏干 → 提前触发提醒
4. **用户反馈学习**：
   - 用户点击"已完成" → 下次提醒正常
   - 用户点击"跳过" → 自动延长间隔
   - 用户点击"提前" → 缩短间隔

**推送渠道**：
- 微信服务号模板消息（低成本）
- App推送（更及时）
- 短信（重要提醒，付费）

**动态调整示例**：

| 场景       | 基础提醒 | 环境调整后         |
| -------- | ---- | ------------- |
| 北京冬天，有暖气 | 周一浇水 | 周三浇水（暖气干燥，提前） |
| 南方梅雨季    | 周一浇水 | 下周一（湿度高，延迟）   |
| 夏天35°C   | 周一浇水 | 周五提醒（高温缩短间隔）  |
| 用户拍照土壤偏湿 | 周一浇水 | 跳过本次，下周一再看    |

#### 4.3.4 病害诊断模块

**流程**：
1. 用户拍照 + 描述症状
2. AI 图像分析（病害识别）
3. 大模型理解症状描述
4. 知识库匹配
5. 生成诊断结果+救治方案

**准确率目标**：
- 常见病害 >90%
- 疑难杂症提供专家入口

#### D.3 竞品 SWOT 简评

#### D.4 差异化定位总结

> **现有玩家**：要么"识别工具"，要么"内容社区"，要么"硬件玩具"，没有人系统解决"我家的这盆植物，今天该怎么养"。
>
> **植物管家**：以"AI 个性化养护"为核心，把 **识别 → 计划 → 提醒 → 诊断 → 硬件** 串成闭环，用环境感知 + 土壤识别 + 数据回流建立长期壁垒。
